# Internationalization (i18n) & Localization (l10n)

## Overview
Extract hardcoded user-facing strings into a translation system, add a new
locale, and wire the UI to switch languages — one module or screen at a time,
keeping every locale complete and the suite green. Use this before launching in a
new market, or when a UI grew with English baked into the components.

## Architecture context (read first)
Replace the stack, locale codes, and commands with your own. Re-read every iteration.
- Default locale: `en`; add: `<target-locale>` (e.g. `es`, `fr`, `ja`).
- `src/i18n/` - the translation catalog and locale loader (create if missing).
- `src/ui/` or `src/components/` - the surfaces with hardcoded strings to extract.
- `test/` - run with `<your-test-command>`.
- Locale files: `src/i18n/locales/<code>.json` (or `.po`, `.yml`).

## Phase 1: Set up the i18n foundation
- [ ] **1.1** Choose and integrate the i18n library
  - Pick the stack's standard lib (e.g. `i18next`, `@formatjs/intl`, `vue-i18n`, `gettext`); wire the provider/middleware
  - Verify: a single translated string renders in the default locale; `<your-test-command>` is green
- [ ] **1.2** Establish the locale catalog and loader
  - Create `src/i18n/locales/en.json` with the first few keys; load the catalog by locale code with a fallback to `en`
  - Verify: a missing key falls back to English, not a raw key or a crash

## Phase 2: Extract strings (one surface per task)
- [ ] **2.1** Extract the primary navigation and shell
  - Move nav, header, and footer strings into the catalog; replace literals with `t("key")` calls
  - Verify: the shell renders identically in `en`; no string is left untranslated in those files
- [ ] **2.2** Extract the main screen/dashboard
  - Move that surface's strings to the catalog; keep keys grouped by feature
  - Verify: the screen renders fully from the catalog; `<your-test-command>` green
- [ ] **2.3** Extract remaining screens and forms
  - Continue surface by surface until no hardcoded user-facing literal remains
  - Verify: a grep for common literal patterns in `src/ui/` returns only non-user-facing strings

## Phase 3: Plurals, dates, numbers & the new locale
- [ ] **3.1** Handle plurals and ICU MessageFormat correctly
  - Convert count-based strings to plural-aware messages; verify the right form renders at 0, 1, and many
  - Verify: tests assert the correct plural form per locale
- [ ] **3.2** Localize dates, times, numbers, and currency
  - Route all date/number/currency output through the `Intl` API or the lib's formatter; use the locale's conventions
  - Verify: tests assert formatting differs correctly across locales
- [ ] **3.3** Add the `<target-locale>` translation
  - Create `src/i18n/locales/<target-locale>.json` with every key from `en.json` translated
  - Verify: a completeness check confirms zero missing keys vs `en.json`; the UI renders fully in the new locale

## Phase 4: Wiring, RTL & quality
- [ ] **4.1** Wire locale selection and persistence
  - Detect from URL/cookie/accept-language, allow override, and persist the choice
  - Verify: a test sets each source and the locale resolves in the documented priority order
- [ ] **4.2** Add RTL support if the new locale needs it
  - Set `dir="rtl"` and flip directional layout where the locale requires it
  - Verify: a visual/test check confirms layout direction for the RTL locale
- [MANUAL] **4.3** Native review of the translation
  - A fluent speaker reviews tone, correctness, and context for `<target-locale>`

## Testing Notes
- Run `<your-test-command>` after EVERY task. A green suite must not depend on a specific locale.
- A locale is "complete" only when it has a value for every key in `en.json` — add a completeness check to CI.
- Hardcoded literals slip back in easily; add a lint/test guard for common patterns after extraction.

## Acceptance criteria
1. No user-facing English literal remains in `src/ui/` (verified by grep + tests).
2. Plurals and date/number/currency formatting are locale-aware and tested.
3. `<target-locale>` has a complete catalog (zero missing keys) and renders fully.
4. Locale selection is wired, persisted, and respects the documented resolution order.

## How OCLoop reads this file
- Markers: `- [x]` complete, `- [ ]` pending (executed), `- [MANUAL]` human-only (skipped), `- [BLOCKED: reason]` blocked (skipped).
- It runs one pending task per fresh session, marks it `[x]`, and continues.
- After marking `[x]`, the agent leaves a short indented note beneath it (the strings extracted and the key naming decision) as memory for the next iteration — prose or plain sub-bullets, never `- [ ]`/`- [x]` lines.
- The run ends automatically when no automatable tasks remain; OCLoop appends a `<plan-complete>` summary itself.
